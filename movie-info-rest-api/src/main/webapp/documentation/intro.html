Розпишу деякі визначення

Ресурс - це одиниця сутності. Під ресурсом ми можемо розуміти Персону, Департамент, Заяву і тд. Надалі памятайте про це.

Якщо ви глянете документацію, то побачите, що вона поділена по сутностях(persons, departments). В свою чергу, кожна сутність може містити або 2 методи або 5 методи. :

  • Розпишу які бувають методи.
  • GET множинний.Фактично, можете називати цей метод - пошуком, оскільки в кожній API є можливість фільтрувати цей множинний пошук по певних полях, які описані в документації. Викликавши цей метод, ви отримаєте загальну інформацію по записах з цієї сутності.Там буде:
  •          Загальна кількість записів в базі по даному пошуку (count)
  • Стартова позиція, з якої ви починаєте пошук(offset)
  • Limit записів, які ви отримаєте.Врахуйте, що сервіс не видасть вам більше ніж 500 записів - limit
  • Список ресурсів. Там вже буде список ресурсів, який видасть вам сервіс.
  • GET by ID. За цим методом ви отримаєте детальну інформацію про конкретний ресурс. Там будуть додаткові поля, правда, переважно, вони є технічні і на цьому етапі розвитку проекту не будуть корисні фронтенду. Щоб доступитись до ресурсу по айдішці, потрібно викликати шось на кштакт /resources/{id}, де resource - це назва відповідної сутності
  • Create Resource(Person, Department, etc...).Деякі факти:
  • Вам завжди потрібно створювати ресурс за допомогою POST методу.
  • Вам завжди потрібно буде відправляти JSON на сервер.
  • Ви завжди повинні очікувати http status 201  для перевірки успішності операції.
  • На сервері існує валідація. Якщо ви вставите невалідне значення в POST/PUT запит, то отримаєте http status - 422 і json меседж з переліком полей, які  не є валідні.

Update Resource(Person, Department, etc...).Деякі факти

  • Вам завжди потрібно створювати ресурс за допомогою PUT методу.
  • Практично в всіх випадках, якщо uri множинний запит є,до прикладу /persons , то для обновлення ресурсу вам потрібно добавити до цього uri лише id ресурсу, наприклад /persons/1.Така ж логіка стосується Get   
  • Вам завжди потрібно буде відправляти JSON на сервер.
  • Ви завжди повинні очікувати http status 200  для перевірки успішності операції.
  • На сервері існує валідація. Якщо ви вставите невалідне значення в POST/PUT запит, то отримаєте http status - 422 і json меседж з переліком полей, які не є валідні.
  • Delete Resource. /resources/{id} , де resources - назва відповідної сутності в множині(persons, departments, ...)
  • Щоб видалити, потрібно викликати це uri з методом DELETE
  • Ви повинні отримати статус 204 - No Content
